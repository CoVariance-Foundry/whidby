# Implementation Plan: M9 Report Generation + Feedback Logging

**Branch**: `009-m09-report-generation` | **Date**: 2026-04-05 | **Spec**: `/specs/009-m09-report-generation/spec.md`
**Input**: Feature specification from `/specs/009-m09-report-generation/spec.md`

## Summary

Deliver module M9 as a deterministic report assembly stage that consumes finalized M4-M8 outputs, emits one schema-stable report JSON sorted by opportunity descending, and writes null-safe per-metro feedback logs to Supabase via a dedicated persistence boundary.

## Technical Context

**Language/Version**: Python 3.11+  
**Primary Dependencies**: `pydantic>=2`, existing pipeline types/utilities, existing M2 Supabase client wrapper, stdlib `datetime`/`uuid`/`typing`  
**Storage**: Supabase PostgreSQL (feedback logging), in-memory Python objects for report assembly  
**Testing**: `pytest`, `pytest-asyncio`, fixture-driven contract tests, optional integration tests marked `@pytest.mark.integration`  
**Target Platform**: Linux/macOS CI runners executing Python pipeline modules  
**Project Type**: Backend data-processing module in monorepo (`src/` + `tests/`)  
**Performance Goals**: Deterministic O(n log n) ordering; report assembly remains lightweight compared with upstream collection/scoring stages  
**Constraints**: No score/classification recomputation in M9; preserve emitted report integrity if feedback persistence fails; null-safe outcome fields; no network dependency for unit tests  
**Scale/Scope**: One report per run, one feedback row per ranked metro, covering typical metro batch sizes generated by current M4-M8 pipeline
